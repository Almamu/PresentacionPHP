register_globals PHP 5.3 en adelante

Hasta PHP 5.3 existía una opción a nivel de configuración llamada "register_globals". Esta función permitía que PHP automáticamente creara variables globales basadas en los parámetros pasados por los métodos GET y/o POST de HTTP. Esto genera un gran riesgo de seguridad ya que permite a un usuario malicioso re-emplazar variables que el programador haya declarado en sus scripts PHP. Por ejemplo:

<?php
  if ($username == 'Alexis' && $password == 'hunter2')
  {
    $valid = true;
  }

  if ($valid == true)
  {
    echo 'Bienvenido ' . $username . '!';
  }

Si a este, en principio, simple script lo llamaramos de la siguiente forma:
http://localhost/ejemplo.php?username=Mago&password=ninguna
no habría ningún problema. La comprobación de contraseña se realizaría de forma correcta y no devolvería nada. Sin embargo, si este script se llamara con un parámetro más, podríamos saltarnos esta comprobación:
http://localhost/ejemplo.php?username=Mago&password=ninguna&valid=1
De esta forma el script anterior mostraría por pantalla "Bienvenido Mago".

PHP OOP moderno PHP 5 en adelante

  • Acceso al código de ejemplo 1
  • Acceso al código de ejemplo 2

Los objetos eran tratados de forma muy distinta. Todos heredaban de stdClass y eran copiados al ser pasados por parámetro. Desde PHP 5.0 esto ha cambiado y los parámetros de tipo object son pasados por referencia automáticamente. Esto también introdujo mejoras de rendimiento y uso de memoria. Adicionalmente se añadió soporte para "Determinación de tipos", que permite añadir un tipo a los parámetros de funciones. Este sistema fue posteriormente extendido con PHP 5.1, con soporte para el tipo array; PHP 5.4, con soporte para el tipo callable; y con PHP 7.0 con soporte para tipos escalares (int, bool, float, string).

Sobre-escritura de métodos PHP 5.1 en adelante

  • Acceso al código de ejemplo 1
  • Acceso al ejemplo de excepción 1

Al usar herencia las funciones han de tener el mismo número de parámetros y tipos, al igual que el tipo de retorno ha de ser el mismo. Desde PHP 7.2 existe una excepción a esta regla, la cual permite eliminar el tipo de retorno o el tipo de alguno de los parámetros para permitir cualquier tipo. Esto se conoce como contra-varianza de tipos.

En adicción a la "Determinación de tipos" introducida en PHP 5, se ha a&ntile;adido soporte gradualmente a usar tipos en otros entornos de PHP, como es el caso de PHP 7.4, con el soporte de tipos para miembros de clases; PHP 7.2, con el tipo "object" que engloba a todas las clases; PHP 7.1, con el soporte para nullables y retornos void; PHP 7.0 con soporte para tipar el retorno de funciones. Esto también ha llevado a la reserva de distintas palabras extra que podrían ser usadas en el futuro: resource, mixed y numeric

Soporte finally en try-catch PHP 5.5 en adelante

El soporte para los bloques finally fue introducido en PHP 5.5, permitiendo ejecutar un código indistintamente si un try-catch ha detectado o no una excepción dentro del try. Esto es especialmente útil para cerrar ficheros, conexiones a base de datos, etc...

Excepciones PHP 5 en adelante

A partir de PHP 7.1 se ha introducido también la opción de capturar más de una excepción en un mismo bloque catch, al estilo Java.

¿Qué le depara el futuro a PHP?

Unión de tipos en parámetros

Propuesto por Nikita
Ha sido propuesto como una extensión al soporte de tipos de PHP, para permitir que una función pueda recibir un mismo parámetro de distintos tipos. Por ejemplo: 

<?php
  function suma (int|float $numero1, int|float $numero2)
  {
    return $numero1 + $numero2;
  }

  echo suma (5, 5);
La salida sería: 
10

Forzar declaración antes del uso de variables

Propuesto por Sara Golemon y Andreas Braun
El objetivo de esta propuesta es forzar que las variables sean declaradas antes de su primer uso, generando un error fatal en caso de que esta regla no se siga. De esta forma se evitarín situaciones de resultados inconsistentes al usar variables no declaradas.

Soporte para anotaciones al estilo java

Propuesto por Michał Brzuchalski
Esta propuesta permitiría generar y leer anotaciones sobre funciones, permitiendo acceso más rápido y sencillo a atributos, especialmente útil para sistemas como Doctrine que hacen un extensivo uso de estas para especificar como una clase se enlaza con una tabla. Actualmente se utilizan comentarios que se han de interpretar de forma manual, lo cual los hace realmente lentos en comparación al uso de anotaciones, las cuales formatín parte del lenguaje y estarín integradas en el intérprete de PHP.
Situación actual: 

<?php
  use ORM\Entity;
  use ORM\Id;
  use ORM\Column;
   
  /**
   * @Entity
   * @Table ("foo")
   */
  class Foo
  {
      /**
       * @Id
       * @Column ("id", "uuid")
       */
      private $id;
  }
Uso propuesto: 
<?php
  use ORM\Entity;
  use ORM\Id;
  use ORM\Column;
   
  [@Entity]
  [@Table("foo")]
  class Foo
  {
      [@Id]
      [@Column("id", "uuid")]
      private $id;
  }
